# Git 提交规范

本项目采用 Conventional Commits，提交信息以中文描述变更内容。

## 1. 基本格式

```txt
<type>(<scope>): <中文摘要>
```

可选扩展：

```txt
<type>(<scope>): <中文摘要>

<正文，可选>

<页脚，可选>
```

## 2. 字段说明

- `type`: 提交类型，使用英文关键字
- `scope`: 影响范围，使用模块名（可选，但推荐）
- `中文摘要`: 一句话说明这次改动做了什么

## 3. type 取值

- `feat`: 新功能
- `fix`: 问题修复
- `refactor`: 重构（不改变外部行为）
- `style`: 样式或格式调整（不改业务逻辑）
- `docs`: 文档变更
- `test`: 测试相关
- `chore`: 构建、依赖、脚本、工具链等杂项

## 4. 摘要要求

- 使用中文，简洁明确，建议 30 字以内
- 使用祈使句/陈述句均可，但避免口语化
- 不要以句号结尾
- 聚焦“做了什么”，不要写“修了一些问题”这类模糊描述

## 5. scope 建议

常用范围示例：

- `upload-queue`
- `file-toolbar`
- `file-explorer`
- `disk-api`
- `i18n`
- `backend`
- `frontend`

## 6. 破坏性变更

有破坏性变更时，在 `type` 后加 `!`：

```txt
feat(upload-queue)!: 调整上传状态字段定义
```

并在正文或页脚补充迁移说明。

## 7. 示例（推荐）

```txt
fix(upload-queue): 仅在全部完成且无失败任务时自动关闭队列
feat(file-toolbar): 队列按钮增加待处理数量徽标
style(upload-queue): 修复状态文案换行问题
docs(readme): 补充上传队列交互说明
chore(frontend): 升级 vue-i18n 到 11.2.8
```

## 8. 一次提交一个目标

- 一个提交只做一类改动，避免把功能、重构、样式混在一起
- 大改动拆成多个可审阅的小提交
- 提交前确保本地检查通过（如类型检查、测试）
